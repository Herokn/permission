# 后端系分文档编写规范

## 1. 文档目的

后端系分文档（系统分析文档）是基于PRD需求文档，将业务需求转换为技术实现方案的文档。该文档用于：

1. **指导AI生成代码**：提供足够详细的技术实现细节，让AI能够生成符合规范的代码
2. **技术方案设计**：明确数据库设计、接口设计、业务逻辑设计等
3. **开发规范遵循**：确保生成的代码符合后端代码开发规范

## 2. 文档结构

后端系分文档应包含以下核心章节：

### 2.1 必需章节

1. **模块概述**：模块基本信息、功能概述、技术栈
2. **数据库设计**：表结构、表关系、数据库操作规范
3. **接口设计**：接口列表、接口详细设计（请求/响应、校验规则）
4. **代码结构设计**：分层架构、包结构设计
5. **业务逻辑设计**：核心业务流程、数据校验规则、权限控制
6. **异常处理设计**：异常码定义、异常处理策略
7. **开发检查清单**：代码规范、功能实现、数据库、接口、测试检查

### 2.2 可选章节

- **日志设计**：如模块有特殊日志需求
- **性能设计**：如模块有性能要求
- **安全设计**：如模块有安全要求
- **单元测试设计**：如需要详细测试用例
- **接口文档**：OpenAPI文档说明

## 3. 从PRD到系分文档的转换指南

### 3.1 数据库设计转换

**PRD中的信息**：
- 数据模型需求（Data Model）章节
- 字段名、类型、必填、说明

**系分文档中需要补充**：
- 数据库字段类型（VARCHAR长度、BIGINT等）
- 索引设计（主键、唯一索引、普通索引、联合索引）
- 约束说明（唯一性约束、外键约束等）
- 字段设计说明（设计决策、特殊处理）

**转换示例**：

PRD中：
```
| 字段名 | 类型 | 必填 | 说明 | 示例 |
| id | String | 是 | 数据集唯一标识 | "ds_123456" |
```

系分文档中：
```
| 字段名 | 类型 | 长度 | 是否必填 | 默认值 | 说明 | 示例值 |
| id | BIGINT | - | 是 | AUTO_INCREMENT | 主键ID | 1 |

索引设计：
PRIMARY KEY (id)
```

### 3.2 接口设计转换

**PRD中的信息**：
- 接口需求（API Requirements）章节
- 接口路径、输入参数、输出结果、校验逻辑

**系分文档中需要补充**：
- HTTP方法（GET/POST/PUT/DELETE）
- 请求参数详细说明（Query参数、Path参数、请求体）
- 响应结果详细结构（成功响应、分页响应、错误响应）
- 错误码定义（按照异常码规范）
- 权限要求

**转换示例**：

PRD中：
```
- **接口路径**：`GET /api/dataset/list`
- **输入参数**：
  - `workspaceId`（查询参数）：工作空间ID，String类型，必填
```

系分文档中：
```
**接口路径**：`GET /api/v1/dataset/list`

**Query参数**：
| 参数名 | 类型 | 必填 | 说明 | 示例值 |
| workspaceId | String | 是 | 工作空间ID | "ws_123456" |

**错误响应**：
| HTTP状态码 | 错误码 | 错误信息 | 说明 |
| 400 | 122001 | 工作空间ID不能为空 | 参数校验失败 |
```

### 3.3 业务逻辑设计转换

**PRD中的信息**：
- 核心逻辑（Logic）章节
- 业务规则、处理逻辑、异常处理

**系分文档中需要补充**：
- 详细的处理步骤（输入、处理、输出、异常）
- 代码层面的实现逻辑
- 事务管理
- 数据流转过程

**转换示例**：

PRD中：
```
**业务规则 1：用户创建流程**
- **触发条件**：系统管理员在管理后台点击"新增用户"按钮
- **处理逻辑**：
  1. 校验必填字段
  2. 校验username唯一性
  3. 密码加密
```

系分文档中：
```
**处理步骤**：

1. **参数校验**：
   - 输入：CreateUserDTO对象
   - 处理：使用@Valid注解校验，检查username、password、roles是否为空
   - 输出：校验通过或抛出参数异常
   - 异常：400 - 参数校验失败

2. **唯一性校验**：
   - 输入：username
   - 处理：调用UserDAO.selectByUsername()查询，包含deleted=1的记录
   - 输出：用户不存在或已存在
   - 异常：409 - 用户名已存在

3. **密码加密**：
   - 输入：明文密码
   - 处理：使用BCrypt加密（强度因子12）
   - 输出：加密后的密码哈希值
```

### 3.4 代码结构设计

PRD中没有直接信息，需要根据功能设计：
- 确定需要哪些Controller、Manager、Service、DAO
- 确定DTO、VO的结构
- 确定包路径和类名

**设计原则**：
- 遵循后端代码开发规范的分层架构
- 遵循命名规范
- 职责单一原则

## 4. 关键设计要点

### 4.1 数据库设计要点

1. **字段类型选择**：
   - ID使用BIGINT自增
   - 字符串根据长度选择VARCHAR
   - 时间使用DATETIME
   - 布尔值使用TINYINT(1)
   - 枚举使用ENUM或VARCHAR+应用层校验

2. **索引设计**：
   - 主键必须有主键索引
   - 唯一字段必须有唯一索引
   - 查询条件字段建立普通索引
   - 联合查询建立联合索引

3. **逻辑删除**：
   - 统一使用deleted字段（TINYINT，0=未删除，1=已删除）
   - 唯一性校验需考虑已删除记录

### 4.2 接口设计要点

1. **RESTful规范**：
   - GET：查询
   - POST：创建
   - PUT：更新
   - DELETE：删除

2. **统一响应格式**：
```json
{
  "code": 200,
  "message": "操作成功",
  "data": {}
}
```

3. **错误码规范**：
   - 按照业务域代码+异常类型+具体错误码的格式
   - 参考后端代码开发规范中的异常码规范

4. **参数校验**：
   - 使用@Valid注解
   - 使用javax.validation注解（@NotBlank、@Length等）

### 4.3 业务逻辑设计要点

1. **事务管理**：
   - 涉及多表操作必须使用事务
   - 事务注解放在Manager层

2. **异常处理**：
   - 业务异常使用BusinessException
   - 参数异常使用IllegalArgumentException
   - 统一异常码和错误信息

3. **权限控制**：
   - 接口权限：使用@PreAuthorize或拦截器
   - 数据权限：在Service层实现

### 4.4 代码结构设计要点

1. **分层职责**：
   - Controller：参数校验、调用Manager、返回响应
   - Manager：业务应用逻辑、事务管理、协调Service
   - Service：核心业务逻辑、领域服务
   - DAO：数据访问

2. **对象转换**：
   - DTO：接口请求/响应对象
   - VO：视图对象，只包含前端需要的字段
   - DO：数据对象，对应数据库表
   - BO：业务对象，领域模型

## 5. AI生成代码的指导原则

### 5.1 提供足够细节

系分文档需要提供足够的技术细节，包括：
- 具体的字段类型和长度
- 具体的校验规则和错误信息
- 具体的业务逻辑处理步骤
- 具体的异常处理方式

### 5.2 遵循开发规范

系分文档必须明确遵循后端代码开发规范：
- 包命名规范
- 类命名规范
- 方法命名规范
- 代码格式规范
- 异常处理规范
- 日志规范

### 5.3 明确设计决策

对于关键的设计决策，需要在系分文档中说明：
- 为什么这样设计
- 有什么考虑
- 有什么限制

## 7. 使用示例

### 7.1 基于PRD生成系分文档的步骤

1. 阅读PRD文档，理解业务需求
2. 提取关键信息：
   - 数据模型需求 → 数据库设计
   - 接口需求 → 接口设计
   - 核心逻辑 → 业务逻辑设计
3. 补充技术细节：
   - 数据库字段类型、索引
   - 接口请求/响应结构
   - 业务逻辑处理步骤
4. 设计代码结构：
   - 确定需要的类和方法
   - 确定包路径和类名
5. 完善异常和日志：
   - 定义异常码
   - 设计日志记录点
6. 检查清单验证：
   - 使用开发检查清单验证完整性

## 8. 常见问题

**Q1: PRD中没有明确的数据库字段类型怎么办？**

A: 根据业务需求和最佳实践确定：
- 字符串：根据实际长度选择VARCHAR长度
- 数字：根据数值范围选择INT/BIGINT
- 时间：统一使用DATETIME
- 布尔：使用TINYINT(1)

**Q2: 如何确定需要哪些Service和Manager？**

A: 根据业务复杂度：
- 简单CRUD：只需要Manager和DAO
- 复杂业务：需要Service处理核心逻辑，Manager协调
- 跨模块：需要Facade提供门面接口

**Q3: 异常码如何分配？**

A: 参考后端代码开发规范：
- 业务域代码：根据模块所属业务域
- 异常类型：根据异常性质（1-用户，2-数据，3-系统，4-权限，5-业务规则）
- 具体错误码：按顺序分配，记录在文档中
