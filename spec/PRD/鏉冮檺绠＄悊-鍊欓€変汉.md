# 一、候选人版

## 技术任务：统一权限管理平台 MVP（前后端）

### 1. 背景

建设统一权限中心（Permission Center），为多个业务系统提供统一的权限管理与鉴权能力。

本系统采用**仅权限点模型**：权限中心只管理 `permissionCode`，不直接管理菜单、页面或资源实体。业务系统通过 `permissionCode` 控制菜单、按钮、接口等权限。同时，系统需要支持**多项目（projectId）维度的权限隔离**，并支持**角色授权 + 用户直接授权 + 用户直接排除权限（DENY）**。

本任务希望你实现一个**最小可用的权限管理平台 MVP**，包含基础后端服务和管理后台。

------

## 2. 任务目标

请实现一个可运行的权限管理平台，至少满足以下目标：

- 管理权限点
- 管理角色，并为角色分配权限点
- 为用户分配角色
- 为用户直接授予权限或排除权限
- 支持按 `projectId` 维度生效
- 提供统一鉴权接口
- 提供一个简单可用的前端管理界面，用于完成配置和验证

我们更看重：

- 业务抽象能力
- 数据建模能力
- 复杂规则落地能力
- 工程实现质量
- 设计取舍说明

## 4. 业务模型要求

### 4.1 权限点（Permission）

权限点是最小权限单位，每个权限点由唯一 `code` 标识。

建议字段：

- `id`
- `code`
- `name`
- `systemCode`
- `type`
- `parentCode`
- `status`
- `description`

说明：

- `code` 全局唯一
- 支持父子关系，便于形成树结构
- `type` 可自行设计枚举，例如：`MENU`、`PAGE`、`ACTION`
- `status` 支持启用 / 禁用

示例：

- `ORDER_MENU`
- `ORDER_VIEW`
- `ORDER_CREATE`
- `ORDER_APPROVE`

### 4.2 角色（Role）

角色是权限点的集合。

建议字段：

- `id`
- `code`
- `name`
- `roleScope`
- `roleDomain`
- `status`
- `description`

说明：

- `roleScope`
  - `GLOBAL`：全局角色
  - `PROJECT`：项目角色
- `roleDomain`
  - `APP`：业务角色
  - `PERM_CENTER`：权限中心管理角色

### 4.3 角色-权限点关系（RolePermission）

用于实现角色与权限点的绑定。

建议字段：

- `roleId`
- `permissionCode`

### 4.4 用户-角色关系（UserRole）

用于为用户分配角色。

建议字段：

- `userId`
- `roleId`
- `projectId`

规则：

- `projectId = NULL` 表示全局角色
- `projectId = 某项目ID` 表示该角色只在该项目内生效

### 4.5 用户直接权限（UserPermission）

用于为用户直接授予或排除权限。

建议字段：

- `userId`
- `permissionCode`
- `effect`
- `projectId`

其中 `effect` 取值为：

- `ALLOW`
- `DENY`

规则：

- `DENY` 优先级最高
- `projectId = NULL` 表示全局生效
- `projectId = 某项目ID` 表示仅该项目生效

以上模型与规则来自统一权限中心 PRD 的核心设计。

## 5. 权限判断规则

请实现统一鉴权接口：

```http
POST /authz/check
```

请求示例：

```json
{
  "userId": "U1001",
  "permissionCode": "ORDER_APPROVE",
  "projectId": "P1"
}
```

返回示例：

```json
{
  "allowed": true
}
```

### 5.1 鉴权优先级

对于 `(userId, permissionCode, projectId)`，按以下顺序判断：

1. 用户直接 `DENY`
2. 用户直接 `ALLOW`
3. 用户角色拥有的权限
4. 默认拒绝

### 5.2 `projectId` 规则

鉴权时，需要同时考虑两类数据：

- `projectId = 当前请求 projectId`
- `projectId IS NULL`

也就是说，全局权限在项目中默认生效，除非被项目级或全局级 `DENY` 覆盖。这个优先级与 PRD 中的核心规则一致。

### 5.3 示例

若用户 `U1`：

- 在 `P1` 被授予角色 `PROJECT_MANAGER`
- 该角色拥有权限 `ORDER_APPROVE`

则：

- `check(U1, ORDER_APPROVE, P1) = true`
- `check(U1, ORDER_APPROVE, P2) = false`

若用户又被直接配置：

- `U1 + ORDER_APPROVE + DENY + P1`

则：

- `check(U1, ORDER_APPROVE, P1) = false`

## 6. 后端接口要求

接口路径和入参可以自行设计，但至少应包含以下能力。

### 6.1 权限点管理

- 新增权限点
- 编辑权限点
- 查询权限点列表
- 查询权限树

建议接口：

```http
POST /permissions
PUT /permissions/{id}
GET /permissions
GET /permissions/tree
```

### 6.2 角色管理

- 新增角色
- 编辑角色
- 查询角色列表
- 为角色分配权限点

建议接口：

```http
POST /roles
PUT /roles/{id}
GET /roles
POST /roles/{roleId}/permissions
```

### 6.3 用户授权

- 为用户分配角色
- 为用户直接授予权限
- 为用户直接排除权限
- 查询某用户在指定项目下的授权情况

建议接口：

```http
POST /users/{userId}/roles
POST /users/{userId}/permissions
GET /users/{userId}/authorizations?projectId=P1
```

### 6.4 权限判断

```http
POST /authz/check
```

## 7. 前端页面要求

前端不要求视觉精美，但要求结构清晰、操作完整、可演示。

### 7.1 权限点管理页

需要支持：

- 权限列表查看
- 权限点新增
- 权限点编辑
- 树形结构展示或父子关系展示

### 7.2 角色管理页

需要支持：

- 角色列表
- 创建角色
- 编辑角色
- 为角色分配权限点

建议展示方式：

- 左侧角色列表
- 右侧权限树勾选

### 7.3 用户授权页

需要支持：

- 输入 `userId`
- 选择 `projectId`
- 给用户分配角色
- 给用户直接授予权限
- 给用户直接排除权限
- 能看到当前用户在当前项目下配置了哪些角色和直接权限

### 7.4 权限测试页

需要支持：

- 输入 `userId`
- 输入 `permissionCode`
- 输入 `projectId`
- 点击 `Check`
- 展示结果：`Allowed / Denied`

加分项：

- 显示命中来源，例如：
  - 来自直接授权
  - 来自角色授权
  - 被直接 `DENY` 覆盖
