# 核心设计说明

## 1. 权限模型设计

### 1.1 核心概念

本系统采用**仅权限点模型**（Permission-Only Model），权限中心只管理 `permissionCode`，不直接管理菜单、页面或资源实体。

```
┌─────────────────────────────────────────────────────────────┐
│                      权限判断流程                            │
├─────────────────────────────────────────────────────────────┤
│  1. 用户直接 DENY  ──────────────────────► 拒绝（最高优先级） │
│         ↓                                                   │
│  2. 用户直接 ALLOW ─────────────────────► 允许              │
│         ↓                                                   │
│  3. 用户角色权限  ─────────────────────► 允许/继续          │
│         ↓                                                   │
│  4. 组织角色权限  ─────────────────────► 允许/继续          │
│         ↓                                                   │
│  5. 默认拒绝    ─────────────────────────► 拒绝             │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 三维权限模型

```
              项目维度 (projectId)
                   │
                   ▼
           ┌───────────────┐
           │   PROJECT 1   │
           │  ┌─────────┐  │
           │  │ ALLOW   │  │
           │  │ DENY    │  │
用户维度    │  └─────────┘  │
(userId) ───►├───────────────┤
           │   PROJECT 2   │
           │  ┌─────────┐  │
           │  │ (无权限)│  │
           │  └─────────┘  │
           └───────────────┘
                   │
                   ▼
              权限维度
          (permissionCode)
```

## 2. 鉴权规则详解

### 2.1 优先级规则

| 优先级 | 来源 | 说明 | 示例 |
|--------|------|------|------|
| 1 | 用户直接 DENY | 最高优先级，覆盖一切 | 用户被禁止访问 P1 的订单审批 |
| 2 | 用户直接 ALLOW | 直接授权，无需角色 | 用户被授予 P1 的订单审批 |
| 3 | 用户角色权限 | 通过角色间接获得 | 用户在 P1 拥有项目经理角色 |
| 4 | 组织角色权限 | 通过组织继承 | 用户属于某组织，组织拥有角色 |
| 5 | 默认拒绝 | 无匹配则拒绝 | 未找到任何授权记录 |

### 2.2 projectId 规则

```
场景 1：全局权限
┌─────────────────────────────────────────┐
│ 用户 U1 在全局范围被授予 ORDER_MANAGER  │
│                                          │
│ check(U1, ORDER_VIEW, P1)  → ✅ 允许    │
│ check(U1, ORDER_VIEW, P2)  → ✅ 允许    │
│ check(U1, ORDER_VIEW, P3)  → ✅ 允许    │
└─────────────────────────────────────────┘

场景 2：项目级权限
┌─────────────────────────────────────────┐
│ 用户 U1 仅在 P1 被授予 ORDER_MANAGER    │
│                                          │
│ check(U1, ORDER_VIEW, P1)  → ✅ 允许    │
│ check(U1, ORDER_VIEW, P2)  → ❌ 拒绝    │
└─────────────────────────────────────────┘

场景 3：混合权限
┌─────────────────────────────────────────┐
│ 用户 U1：                                │
│   - 全局：ORDER_VIEWER（仅查看）         │
│   - P1：ORDER_MANAGER（含审批）          │
│                                          │
│ check(U1, ORDER_APPROVE, P1)  → ✅ 允许 │
│ check(U1, ORDER_APPROVE, P2)  → ❌ 拒绝 │
│ check(U1, ORDER_VIEW, P1)     → ✅ 允许 │
│ check(U1, ORDER_VIEW, P2)     → ✅ 允许 │
└─────────────────────────────────────────┘
```

### 2.3 DENY 覆盖规则

```
场景：DENY 优先级最高
┌─────────────────────────────────────────┐
│ 用户 U1 配置：                           │
│   - P1 角色：PROJECT_MANAGER（有审批权） │
│   - P1 直接权限：ORDER_APPROVE = DENY   │
│                                          │
│ check(U1, ORDER_APPROVE, P1) → ❌ 拒绝  │
│ 原因：被直接 DENY 覆盖（项目 P1）        │
└─────────────────────────────────────────┘
```

### 3.2 核心表说明

| 表名 | 说明 | 关键字段 |

## 4. 设计取舍

### 4.1 为什么采用仅权限点模型？

**选择：** 仅管理 permissionCode，不管理菜单/页面/资源实体

**理由：**
1. **解耦**：业务系统可自由决定如何使用权限点（控制菜单、按钮、接口等）
2. **灵活**：不同业务系统可以有不同的权限实现方式
3. **简洁**：权限中心职责单一，降低复杂度
4. **通用**：适配各类业务场景，不需要预定义资源类型

### 4.2 projectId 的设计选择

**选择：** 使用字符串类型，支持 NULL 表示全局

**理由：**
1. **灵活性**：字符串类型可适配不同业务系统的项目ID格式
2. **简洁**：NULL 表示全局，无需额外的 is_global 字段
3. **查询效率**：直接在索引中处理 NULL 值

**查询逻辑：**
```sql
-- 查找用户在项目 P1 或全局的角色
SELECT * FROM user_role 
WHERE user_id = 'U1' 
AND (project_id = 'P1' OR project_id IS NULL)
```

### 4.3 DENY 优先级设计

**选择：** DENY 具有最高优先级，覆盖一切 ALLOW

**理由：**
1. **安全优先**：当存在争议时，默认拒绝更安全
2. **审计需求**：可以明确禁止特定用户的特定权限
3. **例外处理**：可以临时禁止某用户的权限而不影响其角色配置

### 4.4 角色Scope 和 Domain 的区分

**roleScope（角色范围）：**
- `GLOBAL`：全局角色，跨项目生效
- `PROJECT`：项目级角色，需要在项目上下文中使用

**roleDomain（角色域）：**
- `APP`：业务系统角色，用于具体业务
- `PERM_CENTER`：权限中心管理角色，用于系统管理

**区分目的：**
- roleScope 决定角色的生效范围
- roleDomain 区分角色的业务归属
- 两者正交，可以组合出 4 种角色类型

### 4.5 未实现功能的考量

| 功能 | 状态 | 原因 |
|------|------|------|
| 数据权限 | 未实现 | 复杂度高，需求多样化，建议在业务层实现 |
| 动态权限 | 未实现 | 需要规则引擎支撑，超出 MVP 范围 |
| 权限继承 | 部分实现 | 组织权限继承已实现，权限点继承未实现 |
| 批量操作 | 部分实现 | 批量鉴权已实现，批量授权待实现 |

## 5. 扩展性设计

### 5.1 支持的扩展点

1. **新增权限类型**：type 字段支持扩展
2. **新增角色范围**：roleScope 支持扩展（如 DEPARTMENT）
3. **新增角色域**：roleDomain 支持扩展
4. **新增效果类型**：effect 支持扩展（如 READ_ONLY）

### 5.2 性能优化预留

1. **缓存层**：预留了 AuthzCacheService 接口
2. **索引设计**：关键字段已建立索引
3. **批量查询**：支持批量鉴权接口

---

## 6. 安全考量

### 6.1 JWT 认证

- 使用 HS256 算法签名
- Access Token 有效期 2 小时
- Refresh Token 有效期 7 天

### 6.2 权限校验

- 所有接口默认需要认证
- 默认拒绝：无授权时拒绝访问
- 审计日志：记录所有敏感操作

### 6.3 数据隔离

- 支持 projectId 维度的数据隔离
- 支持 userId 维度的权限隔离
- 支持组织架构维度的权限隔离
